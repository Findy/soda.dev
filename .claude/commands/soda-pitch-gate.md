# Pitch Gate（企画価値審査）

このコマンドは、**執筆前に企画の価値を審査**し、「当たり前すぎる記事」「情報密度が薄い記事」を書き始める前に止めることを目的としています。

## 使い方

### パターン1: 番号を指定する場合

```
/soda-pitch-gate 123
/soda-pitch-gate #123
/soda-pitch-gate 6801883189125703771
/soda-pitch-gate 6801883189125703771.md
```

引数に数字を指定すると、次の順序で処理します：

#### 1-1. 長い数字（はてなブログのエントリーID）の場合
- 数字が10桁以上の場合、はてなブログのエントリーIDとして扱う
- `draft_entries/<エントリーID>.md` ファイルの存在を確認
- ファイルが存在すれば、その内容から企画情報を抽出して審査

#### 1-2. 短い数字（GitHub issue番号）の場合
- 数字が10桁未満、または `#` 付きの場合、GitHub issue番号として扱う
- `gh issue view <issue番号> --json title,body,author,labels --repo findy/soda.dev` を実行
- issueのbodyから企画情報を抽出して審査

### パターン2: 引数なしで実行する場合

```
/soda-pitch-gate
```

引数なしで実行された場合、あなた（Claude）は次の順序で企画情報を探してください：

#### 2-1. ブランチとPR・issueの紐づきを確認

次の手順で企画情報を探してください：

1. `git branch --show-current` を実行して現在のブランチ名を取得
2. `gh pr list --head <ブランチ名> --json number,body,closingIssuesReferences --repo findy/soda.dev` を実行してPRを検索
3. PRが見つかった場合、次の順序でissue番号を特定：
   - `closingIssuesReferences` 配列に値があればその最初のissue番号を使用
   - なければ、PRの`body`から `#数字` のパターンを正規表現で抽出（例: `該当Issue - #492` から `492` を抽出）
4. issue番号が特定できたら `gh issue view <issue番号> --json title,body --repo findy/soda.dev` を実行
5. issueの`title`と`body`から企画情報を抽出して審査を実行

#### 2-2. draft_entriesの変更を確認

2-1でissueが見つからなかった場合、次の手順を実行：

1. `git status --short draft_entries/` を実行して変更されたファイルを検索
2. 変更があるdraft_entriesファイル（`.md`）が見つかったら、Readツールで内容を読み込む
3. 記事のタイトル、見出し、本文から企画情報（想定読者、主張、新規性の根拠）を推定
4. 推定した情報をもとに審査を実行

#### 2-3. 新規企画として扱う

2-1と2-2のいずれも該当しない場合、**パターン3（新規企画）として扱います**。
詳細は下記の「パターン3: 新規企画として実行する場合」を参照してください。

### パターン3: 新規企画として実行する場合

次のいずれかのコマンドで、**新規企画として対話形式**で企画情報を収集します：

```
/soda-pitch-gate new
/soda-pitch-gate 新規
```

このコマンドを実行すると、パターン2-3と同様に、次の質問を順番に行います：

```
1. 仮でいいのでタイトル案や伝えたいイメージを一言で言うと？

2. [1の内容を踏まえて選択肢の順番を変化させる。近そうな職種ほど上へ] 読者のエンジニア職種は？

   個別に選択する場合は次から選んでください（複数可）：

   0) おすすめ: [タイトルから推定される職種を列挙]
   1) バックエンドエンジニア
   2) フロントエンドエンジニア
   3) SRE/インフラエンジニア
   4) データエンジニア
   5) プロダクトマネージャー
   6) エンジニアリングマネージャー
   7) モバイルエンジニア
   8) プロダクトデザイナー
   9) PdM
   10) その他

3. 読者のスキル感は？ミドル？シニア？
   [1-2の内容を踏まえて、1〜3の選択肢を出しながら、それらのレベルのイメージを別の言葉で補足] 
   1) ジュニア（1-3年目）
   2) ミドル（4-7年目）
   3) シニア（8年目以上）

4. 役職はある？
   1) 役職なし
   2) EM
   3) Tech Lead
   4) VPoE
   5) CTO
   6) その他

5. 主張したいことはある？
   [1〜4の内容を踏まえた例示を5つほど行う]

6. [1〜5の回答を踏まえて] この記事でしか読めない新規性は？
   例:
   - Findy固有のプロジェクト事例
   - 独自の判断フレームワーク
   - 具体的な数値データ
   - 失敗談・成功談のディテール
   [具体的な例示を5つほど行う]

7. [6の回答を踏まえて] その新規性を保持するための具体的な材料や取り組みやファクトは？
   例:
   - 実際のプロジェクト名や時期
   - 削減した工数・時間の数値
   - チームメンバーの声
   - Before/Afterの比較データ
   - 判断基準のチェックリスト
   [具体的な例示を5つほど行う]
```

また、すでに企画情報がまとまっている場合は、次の形式で情報を提供することもできます：

```
/soda-pitch-gate

- タイトル案（仮）: 〇〇
- 想定読者: [スキル感][職種][役職]
- 1文の主張: 〇〇
- 新規性の根拠: 〇〇
- 具体を出せる材料: 〇〇
```

# 審査内容

次の項目を審査し、**PASS/FAIL判定**を行います。

## 1. 非自明性（Non-obviousness）

**審査観点:**
- その主張は業界の上位層（ハイスキル層）にとって"既知"ではないか？
- 読後に「それ知ってる」で終わらないか？

**FAIL判定例:**
- 「読書が大事」「コミュニケーションが大事」「テストが大事」みたいな"正論単体"で終わる
- 一般的なベストプラクティスを述べているだけ

**PASS例:**
- 「AI-DLCでドキュメントレビューを落とさないための読みの技術」（具体的な手順に落とし込んでいる）
- 「30ドキュメントを捌くためのスキャン法」（再現可能な型を提示）

---

## 2. 独自性（Originality）

**審査観点:**
- その記事でしか得られない要素があるか？
- 具体的な一次体験のディテールがあるか？

**独自性の要素（最低2つ必要）:**
- [ ] 具体的なディテール（数値、時間、失敗の詳細）
- [ ] ログ・計測結果・比較データ
- [ ] 現場で再現できる手順（チェックリスト、テンプレート、観点リスト）
- [ ] 失敗談・苦労話
- [ ] Before/Afterの比較
- [ ] オリジナルの図解・フロー
- [ ] コード例・設定例

**FAIL判定例:**
- 体験はあるが、結論が一般論に回収されている
- 「おすすめは何でもいい」など、情報密度が薄い
- 既存の記事・書籍・公式ドキュメントを焼き直しただけ

---

## 3. 実務転用性（Actionability）

**審査観点:**
- 読者が"明日から変えられる行動"が1つ以上あるか？

**具体的なアクション例:**
- レビュー観点リスト
- 読み方・学び方の手順
- プロンプトテンプレート
- 評価指標・計測方法
- チェックリスト
- 設定ファイル・コードスニペット

**FAIL判定例:**
- 気づき・感想で終わり、手順がない
- 「〜を意識しましょう」だけで終わる
- 「試してみてください」だけで、何を試すか不明

---

## 4. 想定読者適合（Audience Fit）

**審査観点:**
- ターゲット読者が明確か？
- その層にとっての新規性があるか？
- SODAナレッジとして、読者の期待を満たすか？

**FAIL判定例:**
- ターゲットが曖昧で、結局誰にも刺さらない
- ジュニア向けと言いながら、ジュニアにとっても当たり前の内容
- 「初心者向けチュートリアルをやってみた」だけで、独自の工夫がない

---

## 5. "So What?" テスト

**審査観点:**
- 「だから何？」と聞かれたとき、答えられるか？
- 誰の何が変わるのか明確か？

**FAIL判定例:**
- 「読解力は大事です」→ 誰でも知っている
- 「AIを使うと便利です」→ 当たり前
- 「コミュニケーションが大切です」→ 既知

**PASS例:**
- 「AI生成ドキュメントのレビュー時間を30分から5分に短縮した手順」
- 「レビュー手戻り率を40%削減した質問テンプレート」

---

# 審査フォーマット（機械的判定）

次の条件を**すべて満たす場合のみPASS**とする：

## 必須条件（1つでも満たさないとFAIL）

- [ ] 「1文の主張」が一般論ではない
- [ ] 「新規性の根拠」が2つ以上ある
- [ ] 「明日できるアクション」が1つ以上ある
- [ ] 想定読者が明確（ジュニア/ミドル/シニアのいずれか）
- [ ] "So What?"に対する答えがある

---

```markdown

- **タイトル案**: [タイトル]
- **想定読者**: [どんなことに関心がある/困っている][ミドル/シニア][エンジニア職種][役職]
- **1文の主張**: [主張]
- **新規性の根拠**: [根拠1], [根拠2], ...
- **具体を出せる材料**: [材料]

---

## 審査結果: ✅ PASS / ❌ FAIL

### 1. 非自明性（Non-obviousness）
- 判定: ✅ PASS / ⚠️ 要改善 / ❌ FAIL
- 理由: [理由]

### 2. 独自性（Originality）
- 判定: ✅ PASS / ⚠️ 要改善 / ❌ FAIL
- 新規性の要素数: [N個]
- 理由: [理由]

### 3. 実務転用性（Actionability）
- 判定: ✅ PASS / ⚠️ 要改善 / ❌ FAIL
- 理由: [理由]

### 4. 想定読者適合（Audience Fit）
- 判定: ✅ PASS / ⚠️ 要改善 / ❌ FAIL
- 理由: [理由]

### 5. "So What?" テスト
- 判定: ✅ PASS / ⚠️ 要改善 / ❌ FAIL
- 理由: [理由]

---

## 総合判定

**結果**: ✅ PASS / ❌ FAIL

**判定理由**:
[PASSの場合] この企画は執筆する価値があります。次のステップに進んでください。

[FAILの場合] この企画は執筆前に見直しが必要です。次の改善が必要です：

### 🔴 必須の改善（これがないとFAIL）
- [改善点1]
- [改善点2]

### 🟡 推奨の改善（あるとより良い）
- [改善点1]
- [改善点2]

### 🟢 できれば改善（あればさらに強くなる）
- [改善点1]
- [改善点2]

---

## 改善提案

[具体的な改善提案をここに記載]

例:
- タイトルを「読書が大事」から「AI-DLCでドキュメントレビューを落とさないための読みの技術」に変更
- 新規性の根拠として「30ドキュメントを捌くためのスキャン法（段落単位・要点抽出の型）」を追加
- アクションとして「レビュー時間、手戻り率などの測定指標」を提示
```

---

# 注意事項

- この審査は**執筆前**に行います。記事を書く前に企画の価値を確認してください。
- FAIL判定を受けた企画は、改善してから執筆を開始してください。
- PASSを受けた企画は、執筆を開始してください。

# Issueコメント投稿（PASS時のみ）

## 実行条件

次の **すべて** を満たす場合に実行する：

- 審査結果が **✅ PASS**
- 審査対象のIssueが特定されている（パターン1-2またはパターン2-1で実行した場合）

## フロー

1. **Issue存在確認**: 審査時に使用したIssue番号を使用
   - Issueがない場合（パターン2-2、パターン3など）: このステップをスキップし、「関連Issueがないためコメント投稿はスキップ」と表示して終了
2. **ユーザーに確認**: 次のメッセージを表示して確認を求める
   ```
   🎉 Pitch Gate PASSです！

   この審査結果をGitHubのIssueにコメントしますか？

   対象Issue: #[Issue番号] [Issue URL]
   ```
3. **実行の場合**: 審査結果全文をIssueコメントとして投稿
   ```bash
   gh issue comment <number> --repo findy/soda.dev --body "$(cat <<'EOF'
   ## Pitch Gate 審査結果

   [審査結果の内容をそのまま貼り付け]
   EOF
   )"
   ```
4. **キャンセルの場合**: 「コメント投稿をスキップしました」と表示して終了

## 出力形式（投稿成功時）

```
✅ Issueに審査結果をコメントしました: [Issue URL]
```
