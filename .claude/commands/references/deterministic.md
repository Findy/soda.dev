# 固定ゲート（Deterministic Gates）

このファイルは、SODA品質レビューの「必ず同じ挙動で実行される部分」を定義する。
コマンド（`/soda-review`）およびSkillの両方から、このファイルを唯一の正として参照する。

---

## 絵文字凡例（全ステップ共通）

| 用途 | 絵文字 | 意味 |
|------|--------|------|
| **判定結果** | ✅ | PASS / OK |
| | ⚠️ | 要注意 / 要修正 |
| | ❌ | FAIL / NG |
| **優先度** | 🔴 | 必須（これがあるとレビュー不可） |
| | 🟡 | 推奨（90点を目指すなら修正） |
| | 🟢 | できれば（軽微、余裕があれば） |

---

## Step 0: 対象の確定（必須）

対象を1行で明示してから開始する。

- ファイルの場合: `Target: draft_entries/xxx.md`
- 外部URLの場合: `Target: https://example.com/article`（WebFetchで取得した旨を記載）

### 外部URLの場合の処理

1. WebFetchツールを使用してURLからコンテンツを取得する
2. 取得したMarkdown/HTMLコンテンツをレビュー対象として扱う
3. 修正案を提示する際は、該当箇所の引用と改善案を提示する（ファイル編集はできないため）

---

## Step 1: Risk Gate（致命傷チェック）

### 文体・テイスト保持の原則（Step 1〜3共通）

修正案を提示する際は、次の原則を**必ず守る**：

1. **最小限の修正**: 問題箇所のみを直し、周囲の文章には手を加えない
2. **語彙の保持**: 筆者が使っている言葉をそのまま使う。言い換えない
3. **文体の保持**: 敬体/常体、カジュアル/フォーマルの度合いを変えない
4. **温度感の保持**: 熱量、主観の強さ、断定の度合いを変えない
5. **構文の保持**: 文の長さ、句読点の打ち方、改行の仕方を変えない

**NG**: 「〜だと思います」→「〜です」（断定度の変化）
**NG**: 「めちゃくちゃ大変だった」→「非常に困難でした」（温度感の変化）
**OK**: 「AWS が」→「AWSが」（スペース削除のみ、語彙・文体は維持）

### 検査項目

**1. 炎上リスク（非人間化・倫理問題）**
- 人を動物に喩える表現（猿、犬、豚、害虫など）
- 人をモノ・機械に喩える表現（歯車、道具、リソースなど）
- 職場語彙（配属、マネジメント、部下など）と人以外の存在の組み合わせ
- タイトル単体で差別的・侮蔑的に誤読されうる構造

**2. 機微情報**
- 画像・スクショに社内情報（Slack、Notion、GitHub、個人名、プロジェクト名）
- ソースコードにAPIキー、トークン、内部URL、IPアドレス
- Findy特有の変数名・関数名がそのまま露出

**3. 比較・批判**
- 他社製品・ツールを下げる表現
- 競合他社名の言及
- 「自慢」「マウント」「上から目線」と受け取られうる表現

**4. 根拠なし断定**
- 数値・効果・統計を根拠なく断定
- 法律・仕様に関する断定で出典がない
- → 検出したら「要出典」ラベルを付ける（勝手に補完しない）

**5. 技術・採用ブランディングリスク（Tech & Employer Branding Risk）**

本ルールは「失敗を書いてはいけない」ことを目的としない。
技術的な挑戦や試行錯誤を **前向きな学びとして伝える** ためのガードである。

この記事は、
**「この会社の技術力を信頼できるか」
「この会社で働きたいと思えるか」**
という視点で読まれることを前提とする。

技術的に正しくても、表現や構造によって
**企業価値・採用ブランドを毀損する内容は公開不可**とする。

#### 5-1. ネガティブ印象を強く与えるタイトル構造

タイトル単体で、失敗・停滞・無力感を想起させないか。

- NG例
  - 「〜しても改善しなかった理由」
  - 「〜が失敗した話」
  - 「〜は意味がなかった」
- OK例
  - 「〜から得た学びと次のアクション」
  - 「〜を通じて分かった設計上のポイント」
  - 「〜の試行錯誤から見えた改善余地」

**判定**: タイトルがネガティブ構造のみで成立している場合 → ❌ **FAIL（修正必須）**

#### 5-2. 投げやり・諦め・後ろ向きな言い切り表現

会社の前向きなバリューと乖離する表現を含んでいないか。

- NG例
  - 「全然分からなかった」
  - 「残念ながら〜に終わりました」
  - 「結局うまくいきませんでした」
- OK例
  - 「当時は手探りの状態だった」
  - 「課題は残るものの」
  - 「理解や環境が追いついていなかった部分があった」

**判定**: 投げやり・諦め表現 → ⚠️ **要修正（トーン調整）**

#### 5-3. 学びや次のアクションが示されない失敗談

失敗を述べているにも関わらず、
**「次に何をするのか」「どう改善するのか」が書かれていない**場合。

- NG例
  - 「〜には至りませんでした」で終わる
  - 「結果として成果は出ませんでした」
- OK例
  - 「〜できませんでした。そこで次は〜を試します」
  - 「この経験から、次に取るべきアプローチが明確になりました」

**判定**: 学び・次のアクションがない失敗談 → ⚠️ **要修正（追記必須）**

#### 5-4. 組織課題・人間関係の生々しい暴露

読者が
**「この会社で働くのは不安だ」
「内部がうまく回っていなさそう」**
と感じる表現を含んでいないか。

- NG例
  - 「チーム間の雰囲気が悪かった」
  - 「周囲を巻き込むことができなかった」
  - 「他部署との連携がうまくいかなかった」
- OK例
  - 「チーム間の連携をより強化する余地があった」
  - 「情報共有の仕組みを改善する必要があった」

**判定**: 組織課題の生々しい暴露 → ❌ **FAIL（表現修正必須）**

#### 5-5. 自虐・卑下・技術的信頼性を損なう表現

読者に
**「この会社の技術力は大丈夫か？」**
と思わせる表現を含んでいないか。

- NG例
  - 「よく分からないまま進めた」
  - 「とりあえず動いたのでヨシ」
  - 「意味がなかった」「無駄だった」
- OK例
  - 「試行錯誤を経て理解を深めた」
  - 「暫定対応後、根本原因の調査を続けた」
  - 「期待した成果には届かなかったが、次につながる知見を得た」

**判定**: 技術的信頼性の毀損 → ❌ **FAIL（表現修正必須）**

#### 検出時の対応ルール（まとめ）

**❌ FAIL（公開不可。修正後に再レビュー必須）**
- ネガティブタイトル構造
- 組織課題の生々しい暴露
- 技術的信頼性を損なう表現

**⚠️ 要修正（表現調整・追記で対応可）**
- 投げやり・諦め表現
- 学びなき失敗談

**例外（FAILとしない）**
失敗の記述であっても、次の3点が明確に示されている場合はFAILとしない：
- 技術的な判断プロセス
- 得られた知見
- 次の具体アクション

### 出力形式

```
## Step 1: Risk Gate

**結果: [✅PASS / ❌FAIL]**

### 検出項目
- ❌ X行目：[問題点]（FAIL相当）
- ⚠️ X行目：[問題点]（要修正）

（検出なしの場合は「検出なし」と明記）

---

### 修正案
#### 👉[X行目]: [修正の概要や方向性]
##### 変更前:
修正前の文章全体を記載し、修正が必要な箇所だけを`修正が必要な箇所`のようにバッククォートで囲む

##### 変更後:
修正後の文章全体を記載し、修正した箇所だけを`修正した箇所`のようにバッククォートで囲む

---
```

※絵文字は抜かさない

**❌FAIL の場合**: ❌項目を列挙して終了。修正後に再度 `/soda-review` を実行するよう案内。
**✅PASS の場合**: Step 1.5へ進む。

## Step 1.5: Lint Gate（機械品質チェック）

### 目的

誤字脱字・表記ゆれ・空白・Markdown崩れなどの**表層事故**を機械的に検出し、
公開後の信頼低下・レビュー差し戻し・恥ずかしい修正対応を防ぐ。

このステップでは**文章の良し悪し・表現改善には踏み込まない**。
あくまで「事故になりうる表層ミス」の検出に限定する。

---

### ルールID設計（必須）

* すべての指摘には **Rule ID を必ず付与**する
* Rule ID 形式: `LINT-<CATEGORY>-<3桁>`
* 例: `LINT-TYPO-001`, `LINT-NORM-002`

---

### 重大度の優先順（固定）

指摘が多い場合は、**次の優先順で上位N件のみ表示**する。

1. **自動FAIL条件に該当するもの**
2. LINT-MD（Markdown）
3. LINT-LINK（リンク）
4. LINT-TERM（用語・固有名詞）
5. LINT-STRUCT（構文）
6. LINT-TYPO（誤字脱字）
7. LINT-NORM（表記ゆれ）

---

### チェック対象（カテゴリ別）

#### LINT-TYPO（誤字脱字）

* LINT-TYPO-001 明らかなタイプミス
* LINT-TYPO-002 助詞の脱落
* LINT-TYPO-003 読み直せば即気づく誤り

##### 誤字脱字辞書（よくあるタイポ）

| 誤表記 | 正表記 | 備考 |
|--------|--------|------|
| おかげて | おかげで | |
| 自由時自在 | 自由自在 | |
| 家に篭りがち | 家に籠りがち | |
| 聞き初めて | 聞き始めて | |
| 〜のの〜 | 〜の〜 | 助詞重複 |
| 残論点として残って | 論点として残って | 重複 |

#### LINT-NORM（表記ゆれ・正規化）

* LINT-NORM-001 同一用語の揺れ（例: GitHub / Github）
* LINT-NORM-002 半角・全角の不統一
* LINT-NORM-003 **【禁則】半角英数字と全角文字の境界スペース**（例: `AWS が`、`v3.0 では`、`ML3 を`）
    * 半角英数字の直後に全角文字、または全角文字の直後に半角英数字が来る場合に挟まる半角スペースは **すべて削除する**（例: `AWS が` → `AWSが`、`レベル N` → `レベルN`、`v2.0 以降` → `v2.0以降`）
    * 例外: コードブロック（``` ``` ）・インラインコード（` ` ）・URL・Mermaid 等のコード内テキストは対象外
    * テーブル区切り `|` の前後の空白、Markdown 構文上必要な空白は対象外

#### LINT-STRUCT（文としての破綻）

* LINT-STRUCT-001 主語と述語が噛み合っていない可能性
* LINT-STRUCT-002 文の途中で意味が切れている可能性

※ LINT-STRUCT は **原則「要判断」扱い**。明確に破綻している場合のみ指摘する。

#### LINT-MD（Markdown）

* LINT-MD-001 見出し記法の崩れ
* LINT-MD-002 コードフェンスの未閉じ
* LINT-MD-003 箇条書きのネスト崩れ

#### LINT-LINK（リンク）

* LINT-LINK-001 URL/リンク記法の崩れ（括弧ミス等）
* LINT-LINK-002 スキーマ欠落（http/httpsなし）

#### LINT-TERM（固有名詞・用語）

* LINT-TERM-001 辞書に定義された固有名詞の誤記

**重要な原則**

* LINT-TERM は **辞書一致ベースのみ**
* 辞書にない語は **触らない**
* AIによる推測・候補提示は禁止
* 確定置換できる場合のみ指摘する

---

### 用語・固有名詞辞書

用語・固有名詞辞書は `.claude/commands/references/lint-dict.md` を参照。

---

### 判定ルール

#### 原則

* 指摘件数・内訳は必ず数値で出す
* 軽微な指摘のみの場合は **PASS** として次へ進む
* ただし、**事故防止のため一部条件では自動FAIL**とする（下記参照）

#### 自動FAIL条件（事故防止）

次のいずれかに該当した場合のみ **❌FAIL** とする。

| 条件 | 理由 |
|------|------|
| LINT-MD-002（コードフェンス未閉じ）が **1件以上** | 表示崩壊 |
| LINT-LINK-001（リンク記法崩れ）が **1件以上** | 読者体験の致命的毀損 |
| LINT-TERM-001（辞書ベース固有名詞誤記）が **1件以上** | ブランド毀損 |
| LINT-TYPO が **100行あたり3件以上** | 品質担保不可 |

※ 行数が取得できない場合は暫定で TYPO合計 **5件以上** を FAIL とする（運用で調整可）

---

### 修正案提示ルール（厳守）

* 提示するのは **最小修正のみ**
  * 誤記の置換
  * スペース削除・追加
  * 記法修正
* **言い換え・表現改善は禁止**
* 判断が分かれる場合は「要判断」と明記する

---

### 出力形式（必須）

```
## Step 1.5: Lint Gate

**結果: [✅PASS / ❌FAIL]**

### 検出サマリ
- 検出件数（合計）: X件
- 種別ごとの内訳
  - LINT-TYPO: A件
  - LINT-NORM: B件
  - LINT-STRUCT: C件
  - LINT-MD: D件
  - LINT-LINK: E件
  - LINT-TERM: F件

### 指摘一覧（上位10件）
- [LINT-XXX-001] X行目: 原文（一行引用） → 置換案（最小修正）
- [LINT-XXX-002] X行目: 原文（一行引用） → 要判断
...
```

※ 自動FAIL条件に該当する指摘は **件数に関係なく全件表示**

---

### ステップ遷移

* **❌FAIL の場合**: 事故防止条件に該当した項目のみ列挙し終了。修正後に `/soda-review` の再実行を案内する。
* **✅PASS の場合**: Step 2（Self Review）へ進む。

## Step 4-1: Issue整合性チェック

### 確認手順

1. PRからissue番号を特定する（`gh pr list --head <ブランチ名> --json number,body --repo findy/soda.dev` でPRを取得し、bodyから `#数字` を抽出）
2. issueが見つかった場合、issueのtitleとbodyを取得
3. issueで宣言した「伝えたいこと」「想定読者」「新規性」が記事に反映されているかを確認

### 確認観点

- [ ] **主張の一致**: issueで宣言した主張が記事の結論と一致しているか
- [ ] **想定読者の一致**: issueで想定した読者層に向けた内容になっているか
- [ ] **新規性の実現**: issueで約束した「この記事でしか読めない要素」が含まれているか
- [ ] **スコープの適切さ**: issueで宣言した範囲から大きく逸脱していないか
