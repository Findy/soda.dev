# 裁量フェーズ

このファイルは、SODAレビューの裁量フェーズ（Step 2, 3, 4-2, 5, 全体サマリ）を定義する。
`/soda-review` から参照される。

---

## Step 2: Self Review（品質チェック）

### 3つの視点でチェック（必須）

各項目に ✅⚠️❌ を付与する。

記事全体を3つの異なる視点で読み直す。1回のチェックでは見落としやすいため、**視点を変えて3回チェック**することが重要。

#### 1回目: 執筆者視点（主張の確認）
- [ ] この記事で最も伝えたいメッセージは何か、一文で言えるか
- [ ] そのメッセージを支える根拠（データ、事例、コード）があるか
- [ ] タイトルと結論が一致しているか

#### 2回目: 読者視点（理解の確認）
- [ ] 前提知識がない人でも理解できるか
- [ ] 「なぜ？」「具体的には？」と思う箇所に答えがあるか
- [ ] 読者が持ち帰れる知見やアクションがあるか

#### 3回目: 批判者視点（リスクの確認）
- [ ] 主張に対する反論や例外を想定しているか
- [ ] 他社製品・競合への批判と取られる箇所はないか
- [ ] 事実として書いている内容に出典・根拠があるか
- [ ] 「自慢」「マウント」「上から目線」と受け取られる表現がないか

---

### チェックリスト

各項目に ✅⚠️❌ を付与する。

**0. 記事の価値**
- [ ] この記事を読まないと得られない情報は何か、一文で言えるか
- [ ] 具体的なエピソード（数字、コード例、失敗談など）が含まれているか

**1. 構造**
- [ ] 目次（`[:contents]`）を入れているか
- [ ] 冒頭で「何の記事か」「何がわかるか」が明確か
- [ ] 記事の最後に採用リンクを入れているか
- [ ] 見出しが具体的か（「1つ目」ではなく内容を示す）

**2. 表記**
- [ ] Findy製品名が正式表記か
- [ ] 技術名の大文字小文字が正しいか（GitHub, JavaScript等）
- [ ] 英単語と日本語の間に不要なスペースがないか
- [ ] 「以下」ではなく「次の」を使っているか

**3. 数値・成果**
- [ ] 改善を主張する場合、Before/Afterの数値があるか
- [ ] 曖昧な表現（「ほぼ100%」「高速になった」）を避けているか

**4. トーン**
- [ ] ネガティブ表現から始まっていないか
- [ ] 砕けすぎた表現を使っていないか

**5. 論理・一貫性**
- [ ] タイトルと内容が一致しているか
- [ ] 導入部で約束したことが本文で果たされているか

**6. 文章表現**
- [ ] 一文が長すぎないか（150文字以内が目安）
- [ ] 誤字脱字はないか

**7. AI的表現の回避**
- [ ] 箇条書きを多用しすぎていないか
- [ ] 太字を多用していないか

**8. 最終確認**
- [ ] 「TODO」などの仮テキストが残っていないか

### 採点基準

| 条件 | 上限点数 |
|------|----------|
| 🔴必須項目（目次・採用リンク・文章破綻）が❌ | 79点 |
| 🟡推奨項目のみ❌/⚠️ | 80〜89点 |
| 全項目✅ | 100点 |

### 減点目安

| 項目カテゴリ | 減点幅 |
|-------------|--------|
| 🔴必須項目（目次・採用リンク・文章破綻） | -10〜-21点（79点上限） |
| 🟡推奨項目（表記・トーン・論理） | -2〜-5点/項目 |
| 🟢軽微（できれば改善） | -1〜-2点/項目 |

### 出力形式

```
## Step 2: Self Review

### チェック結果
（各項目に✅⚠️❌を付けて列挙）

### 採点: XX点 / 100点

**内訳:**
- 目次なし: -10点
- 採用リンクなし: -5点
- 英単語スペース不統一: -2点
- ...
- **合計: XX点**

### 優先度表

| 優先度 | 該当行 | 項目 | 影響 | 減点 |
|--------|--------|------|------|------|
| 🔴 必須 | X | ... | レビュー不可 | -10 |
| 🟡 推奨 | X-Y | ... | 品質向上 | -3 |
| 🟢 できれば | X,Y,Z | ... | 読者体験向上 | -1 |

---
```

**重要**: Risk Gate の結果は再判定しない。このステップは品質チェックのみに集中する。

**90点未満の場合**: Step 3へ進む。
**90点以上の場合**: Step 3をスキップし、Step 4へ。

## Step 3: Fix Plan（改善計画）

### 実行条件

- 原則：**90点未満の場合**実行
- 例外：
  - `--force-fix` 指定時：90点以上でも実行
  - `--no-fix` 指定時：実行しない
  - `--fix-only` 指定時：採点なしで実行（記事から改善点を抽出して作る）

### ルール

🔴必須 → 🟡推奨 → 🟢できれば の順で具体的な書き換え案を提示する。

### 文体・テイスト保持ルール（必須）

**deterministic.md の Step 1 で定義した「文体・テイスト保持の原則」を必ず適用する。**

加えて、次のルールを守る：

- 修正後の文章は、**筆者本人の文体・語彙・テンポ・温度感を最大限維持する**
- 新しい言い回しを発明しない。元の文章に含まれる語彙・構文を優先的に再利用する
- 丁寧になりすぎない／説明的になりすぎない（公式文書化しない）
- 断定の強さ・カジュアルさ・主観度合いを、修正前と同程度に保つ
- 「AIが書いた感」が出る表現（一般論化・抽象化・教科書調）に寄せない

**NG例**:
- 体験談 → 一般論に書き換える
- カジュアルな文 → 無難で硬い文にする
- 主語を「私」から「一般に」に変える
- 「〜してみた」→「〜を実施した」（硬くなる）
- 「めっちゃ」→「非常に」（温度感が下がる）

**OK例**:
- 同じ言い回しを使いながら、語尾や構造だけを調整する
- 文を分ける／つなぐが、語彙は変えない
- スペースや句読点の修正のみで、言葉自体は変えない

### 出力形式

問題、Before、Afterでそれぞれ改行を入れること。

```
## Step 3: Fix Plan

### 🔴 必須修正

#### 👉1. [問題項目名]

[X行目]
[説明/修正の概要や方向性]

##### 変更前:
修正前の文章全体を記載し、修正が必要な箇所だけを`修正が必要な箇所`のようにバッククォートで囲む

##### 変更後:
修正後の文章全体を記載し、修正した箇所だけを`修正した箇所`のようにバッククォートで囲む

---

### 🟡 推奨修正（90点を目指す場合）

#### 👉1. [問題項目名]

[X行目]
[説明/修正の概要や方向性]

##### 変更前:
修正前の文章全体を記載し、 **修正が必要な箇所だけ** を`修正が必要な箇所`のようにバッククォートで囲む

##### 変更後:
修正後の文章全体を記載し、 **修正した箇所だけ**を`修正した箇所`のようにバッククォートで囲む

---

### 🟢 さらに良くする場合

（同様の形式）

---
```

**重要**: このステップは再採点しない。修正案の提示のみに集中する。

---

## テイスト一貫性チェック（内部確認・必須）

Fix Plan の全修正を **仮にすべて適用した前提**で、次を確認する：

- 修正前と比べて、筆者が別人に見えないか
- 語彙・文体・テンポが急に変わっていないか
- 熱量や主観が削がれていないか
- 「正しいがつまらない文章」になっていないか

問題がある場合：
- Fix Plan の修正案を **最小限に書き戻す**
- 構造・安全性・明確さを保ったまま、元の言い回しに寄せ直す

このチェック自体は出力しないが、**全体サマリの「改善ポイント」「次のアクション」に必ず反映する**

## Step 4-2: 記事全体の俯瞰チェック

Step 2の細かいチェックとは別に、**記事を一本の読み物として俯瞰**して確認する。

### 確認観点

- [ ] **論理の流れ**: 文と文、段落と段落のつながりに飛躍がないか
- [ ] **見出しと内容の整合**: 各見出しの下に書かれている内容が見出しと合っているか
- [ ] **テイストの統一**: 冒頭と末尾で温度感・文体が変わっていないか（途中で急に硬くなる、カジュアルになる等）
- [ ] **結論の明確さ**: 読者が読み終えたときに「結局何が言いたかったか」が明確か
- [ ] **導入と結論の呼応**: 冒頭で提起した問いや約束が、結論で回収されているか

### 出力形式

```
## Step 4: 企画価値確認 + 俯瞰チェック

### Issue情報
- Issue: #XXX（または「Issueなし」）
- 当初の主張: [issueから抽出した主張]
- 想定読者: [issueから抽出した想定読者]

### Issue整合性チェック
- 主張の一致: ✅ / ⚠️ / ❌
- 想定読者の一致: ✅ / ⚠️ / ❌
- 新規性の実現: ✅ / ⚠️ / ❌
- スコープの適切さ: ✅ / ⚠️ / ❌

### 俯瞰チェック
- 論理の流れ: ✅ / ⚠️ / ❌
- 見出しと内容の整合: ✅ / ⚠️ / ❌
- テイストの統一: ✅ / ⚠️ / ❌
- 結論の明確さ: ✅ / ⚠️ / ❌
- 導入と結論の呼応: ✅ / ⚠️ / ❌

### 所見
[整合性・一貫性についてのコメント]
```

**Issueが見つからない場合**: 記事単体で「この記事でしか得られない価値」が明確かを確認し、所見に記載する。

## Step 5: フィードバック俯瞰確認

全ステップの結果を踏まえ、**フィードバック全体を俯瞰して**、客観的・一貫性のある内容になっているかを確認してからユーザーに伝える。

### 確認観点

- [ ] **矛盾がないか**: 各ステップの判定結果が互いに矛盾していないか
- [ ] **優先度が適切か**: 指摘事項の優先度（🔴🟡🟢）が妥当か
- [ ] **実行可能か**: 改善提案が具体的で、筆者が実行可能な内容か
- [ ] **バランスが取れているか**: 良い点と改善点のバランスが取れているか
- [ ] **トーンが適切か**: 建設的で前向きなトーンになっているか
- [ ] **点数が妥当か**: 採点が厳しすぎたり甘すぎたりしていないか

### 確認のポイント

1. **客観性**: 個人の好みではなく、ガイドラインに基づいた指摘か
2. **一貫性**: Risk Gate、Self Review、Fix Plan、企画価値確認の結果が整合しているか
3. **有用性**: 筆者にとって本当に役立つフィードバックになっているか

このステップは内部確認のため出力しない。問題があれば全体サマリで調整する。

---

## 全体サマリ（必ず最後に出力）

全ステップの結果を統合し、テイスト一貫性チェック・フィードバック俯瞰確認を踏まえて出力する。

```
## 全体サマリ

### Risk Gate: [✅PASS / ❌FAIL]
### Lint Gate: [✅PASS / ❌FAIL]（検出: N件）
### 点数: XX点 / 100点
### 企画価値: [✅実現 / ⚠️一部未達 / ❌未達]
### レビュー依頼: [✅可能 / ❌修正が必要]

### 良いところ
- [評価できるポイント]

### 改善ポイント
- [修正が必要な箇所と内容]

### 次のアクション
- [具体的に何をすべきか]
```

## Step 6: PRコメント投稿（合格時のみ）

### 実行条件

次の **すべて** を満たす場合に実行する：

- 点数が **90点以上**
- レビュー依頼が **✅可能**
- 現在のブランチに対応するPRが存在する

### フロー

1. **PR存在確認**: `gh pr view --repo findy/soda.dev --json number,url` でPRを取得
   - PRがない場合: このステップをスキップし、「PRがないためコメント投稿はスキップ」と表示して終了
2. **ユーザーに確認**: 次のメッセージを表示して確認を求める
   ```
   🎉 合格ラインに達しました！

   このレビュー結果をGitHubの該当プルリクエストにコメントしますか？

   対象PR: [PR URL]
   ```
3. **実行の場合**: 全体サマリ全文をPRコメントとして投稿
   ```bash
   gh pr comment <number> --repo findy/soda.dev --body "$(cat <<'EOF'
   ## SODA品質レビュー結果

   [全体サマリの内容をそのまま貼り付け]
   EOF
   )"
   ```
4. **キャンセルの場合**: 「コメント投稿をスキップしました」と表示して終了

### 出力形式（投稿成功時）

```
✅ PRにレビュー結果をコメントしました: [PR URL]
```
